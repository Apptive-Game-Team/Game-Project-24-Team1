# Unity C# Detailed Coding Conventions

You MUST strictly follow these conventions when generating or modifying any C# scripts in this project. Reference standards include Microsoft and Google C# guidelines.

## A. Naming & Structure
- **Class / Struct:** `PascalCase` (Use nouns. e.g., `PlayerController`, `MatchResult`).
- **Interface:** Prefix `I` + `PascalCase` (e.g., `IDamageable`).
- **Enum:** `PascalCase` (Use singular nouns. e.g., `WeaponType`, NOT `WeaponTypes`).
- **Methods:** `PascalCase` (Use verbs. e.g., `CalculateScore()`).
- **Events:** Prefix `On` + `PascalCase` (e.g., `OnPlayerDeath`).
- **Booleans:** Prefix with `is`, `has`, or `can` (e.g., `isGrounded`, `hasWeapon`).
- **Local Variables & Parameters:** `camelCase` (e.g., `moveSpeed`).
- **Private Fields:** Prefix `_` + `camelCase` (e.g., `_currentHealth`).
- **Properties:** `PascalCase` (e.g., `CurrentAcs { get; private set; }`).
- **Constants:** `UPPER_SNAKE_CASE` (e.g., `MAX_HEALTH`).
- **UI Components:** Use specific prefixes based on type:
  - Button: `btn_`
  - Text: `txt_`
  - Image: `img_`
  - Slider: `sld_`
  - InputField: `inp_`
- **No Abbreviations:** Do not use arbitrary abbreviations (Use `playerDamage`, NOT `pDmg`).
- **Namespace:** Wrap ALL scripts in the project's specific namespace (e.g., `namespace MyAwesomeGame.Core`).

## B. Code Formatting
- **Braces `{}`:** Use the Allman style. Braces MUST be placed on a new line.
  ```csharp
  // CORRECT
  if (isGrounded)
  {
      Jump();
  }
  ```

## C. Inspector & Access Control
- **Encapsulation:** NEVER use `public` fields just for Inspector exposure.
- **Serialization:** Use `[SerializeField] private` for variables that need to be adjusted by designers in the Unity Editor.
- **Attributes:** Actively use Inspector formatting attributes:
  - `[Header("GroupName")]`: To group related variables.
  - `[Range(min, max)]`: For numeric values that have strict limits.
  - `[Tooltip("Description")]`: To provide context for designers.

## D. Optimization (CRITICAL)
- **No Heavy Calls in Update:** NEVER use `GetComponent<T>()` or `GameObject.Find()` inside `Update()`, `FixedUpdate()`, or `LateUpdate()`. Cache them in `Awake()` or `Start()`.
- **Tag Comparison:** ALWAYS use `gameObject.CompareTag("TagName")`. NEVER use `gameObject.tag == "TagName"`.
- **Garbage Collection (GC) Prevention:**
  - **No Instantiate/Destroy in Hot Paths:** Use **Object Pooling** for frequently created/destroyed entities (e.g., bullets, effects).
  - **No String Mutation in Update:** Do not use string concatenation (`+`) or `.ToString()` inside `Update()`. Update UI text ONLY when the underlying value changes.
  - **No LINQ in Hot Paths:** Avoid `System.Linq` (like `.Where()`, `.ToList()`) during active gameplay loops. Use standard `for` or `foreach` loops.
- **Physics:** ALL Rigidbody manipulations and physics calculations MUST occur inside `FixedUpdate()`, NOT `Update()`.
- **Raycasting:** ALWAYS pass a `LayerMask` to `Physics.Raycast` to limit performance cost. Do not raycast against everything.
- **Coroutines:** Call coroutines using method references, NEVER by string name.
  - Correct: `StartCoroutine(AttackRoutine());`
  - Incorrect: `StartCoroutine("AttackRoutine");`

## E. Documentation (Mandatory Korean Output)
- **XML Comments:** Use XML comments (`/// <summary>`) for all classes and major logic methods.
- **Language Rule:** **ALL comments, tooltips, and explanations MUST be written in KOREAN.** The code itself (variables, methods) remains in English, but the documentation generated by you must be in natural Korean to assist local team members.
